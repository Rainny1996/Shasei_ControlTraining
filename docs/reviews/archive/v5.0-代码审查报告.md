## 二、P0 — 编译阻断缺陷（必须修复）

### BUG-CT-08 `CoachViewModel` 重复 `deinit`

`CoachViewModel.swift` 在同一类中出现了两个 `deinit` 块：

```swift
// 第 140 行
deinit {
    NotificationCenter.default.removeObserver(self)
}
// ...
// 第 205 行
deinit {
    stopTimer()
    audioService.stopAll()
}
```

Swift 规定"一个类至多一个 deinitializer"，上述代码会报
`error: Class 'CoachViewModel' has more than one deinitializer`。
**修复**：合并为单个 `deinit`：

```swift
deinit {
    NotificationCenter.default.removeObserver(self)
    stopTimer()
    audioService.stopAll()
}
```

### BUG-CT-09 `CDTrainingRecord` 缺少 `isPartial` 属性

域模型 `TrainingRecord` 含 `isPartial: Bool`（对应 AC-2.10 强制退出生成部分记录），但 `ControlTrainingModel.xcdatamodeld` 的 `CDTrainingRecord` 实体仅定义了 `id/methodId/date/duration/completionRate/selfRating/notes/mode`，**没有 `isPartial`**。

由于 `codeGenerationType="category"`，Xcode 在构建时根据模型自动生成 `*+CoreDataProperties.swift`，其中不会包含 `isPartial`，因此手写文件中的引用全部失败：

```swift
// CDTrainingRecord+CoreDataClass.swift
self.setValue(record.isPartial, forKey: "isPartial")   // KVC 可编译，但运行时无效
// toDomainModel():
isPartial: isPartial   // ❌ 'CDTrainingRecord' has no member 'isPartial'
```

`DataViewModel.swift` 中的 `cd.isPartial = record.isPartial ?? false` 同样失败。
**修复**：在 `ControlTrainingModel.xcdatamodeld` 的 `CDTrainingRecord` 实体中增补属性：

```xml
<attribute name="isPartial" optional="YES" attributeType="Boolean"
           defaultValueString="NO" usesScalarValueType="YES"/>
```

### BUG-CT-10 `KeychainService` 无 `clearAll()` 方法

`DataViewModel.performDelete()` 调用：

```swift
keychainService.clearAll()   // ❌ 不存在
```

而 `KeychainService` 实际提供的方法是 `clearAllSensitiveData()`。
**修复**：`keychainService.clearAllSensitiveData()`。
**安全补充**：彻底删除数据时还应清除应用密码，否则 `UserDefaults` 被清空后 `isPasswordSet=false` 但 Keychain 中的哈希密码残留，锁屏逻辑虽失效但凭据未销毁——建议同时调用 `SecurityService.shared.removePassword()`。

### BUG-CT-11 `CDCheckInRecord` 无 `status` 属性

`DataViewModel.writeImportedData` 写入：

```swift
cd.status = checkIn.status   // ❌ CDCheckInRecord 模型无 status
```

模型 `CDCheckInRecord` 仅有 `id/date/checkInTime/trainingRecordId`。`CheckInEntry.status` 在域模型与导出逻辑中均无对应概念。
**修复**（二选一）：
- 推荐：删除 `status` 相关代码（`CheckInEntry.status` 字段与 `cd.status` 赋值），导出/导入只保留既有字段；或
- 在模型为 `CDCheckInRecord` 增加 `status` 属性并补齐域模型 `CheckInRecord`。考虑到 AC-8 导出设计无需 `status`，**删除更简洁**。

### BUG-CT-12 `methodId` 类型不匹配

`DataExport/RecordEntry` 将 `methodId` 定义为 `String`：

```swift
struct RecordEntry: Codable {
    let methodId: String   // ❌ 与 Core Data 的 UUID 不一致
}
// collectAllData:
RecordEntry(id: $0.id ?? UUID(), methodId: $0.methodId ?? "", ...)
// $0.methodId 是 UUID?，传入 String 形参 → 类型不匹配
```

**修复**：将 `RecordEntry.methodId` 改为 `UUID`，并相应修改 `collectAllData` 的传参与 `writeImportedData` 的赋值（`cd.methodId = record.methodId`）。

## 三、P1 — 安全缺陷

### SEC-01 4 位纯数字 PIN + 单次 SHA256（可被瞬时爆破）

`SecurityService.hashPassword` 仅做一次 `SHA256(salt ‖ password)`：

```swift
let hashed = SHA256.hash(data: combinedData)
```

而锁屏 `LockScreenView` 使用 4 位纯数字键盘（`enteredPassword.count == 4`）。4 位数字共 1 万种组合，单次 SHA256 在现代设备上可在毫秒级穷举。即便有随机盐，盐值同样存于 Keychain，攻击者在越狱/备份提取后可直接离线爆破。
**建议**：改用密钥派生函数 `PBKDF2`（≥10 万次迭代）或 `Argon2`/`scrypt`；并考虑提高 PIN 长度或在安全场景引入更长密码。属隐私合规（AC-7.x）级别问题。

### SEC-02 `secureDelete` 的 `memset` 无效

```swift
func secureDelete(_ data: Data) {
    var mutableData = data
    let count = mutableData.count
    memset(&mutableData, 0, count)   // ❌ Swift Data 为写时复制值类型
}
```

`Data` 是写时复制的值类型，`&mutableData` 取到的是值语义包装盒，并非底层字节缓冲；`memset` 不会清零实际数据，甚至可能触发 CoW 复制。`Data` 一旦离开作用域即被 ARC 释放，函数无实际安全效果。
**建议**：若需安全擦除，使用 `mutableBytes` 的 `withUnsafeMutableBytes` 在生命周期内清零，或接受 `Data` 无法强制清零并移除该函数（避免"已擦除"的虚假安全感）。

### SEC-03 导入校验逻辑倒置（有效备份被拒绝）

`DataViewModel.performImport`：

```swift
guard !exports.trainingRecords.isEmpty || exports.checkInRecords.isEmpty else {
    throw DataError.invalidBackup
}
```

逻辑分析：当备份**含训练记录**时 `!trainingRecords.isEmpty` 为 `true` → 直接抛错；当**两者皆空**时 `!false || true` 也为 `true` → 抛错。结果：**几乎任何备份都会被拒绝**，仅"无训练记录但含打卡记录"的极小众情况能通过。导入功能实质不可用。
**修复**：仅在两者皆空时拒绝：

```swift
guard !exports.trainingRecords.isEmpty || !exports.checkInRecords.isEmpty else {
    throw DataError.invalidBackup
}
```

## 四、P2 — 正确性与架构缺陷

### ARC-01 字段级 AES-256-GCM 加密从未落地（需求声明与实际不符）

需求/设计声称"敏感字段 AES-256-GCM 加密"。代码中存在 `EncryptedField` 属性包装器与 `DataIntegrityValidator`，但全仓搜索确认二者**均未被任何模型或服务使用**（仅 `SecurityService`/导出流程调用 `CryptoService.encrypt/decrypt`）。即持久化到 Core Data 的笔记/复盘文本等敏感字段实际**未加密**，仅依赖 `DataController` 的文件级保护 `FileProtectionType.complete`（设备锁定时才安全）。
**影响**：设计声明（AES-256-GCM 字段加密）未兑现；`EncryptedField` 与 `DataIntegrityValidator` 为死代码。
**建议**：要么将 `EncryptedField` 应用到 `TrainingRecord.notes` / `ReviewNote.notes` 等敏感字段，要么在文档中明确"仅采用文件级加密"，避免合规误导。

### ARC-02 `CoreDataStack.swift` 为死代码 —— ❌ 已撤回（审查误判）

> **更正（2026-07-11）**：本条经实现方复核与核实，**不成立，予以撤回**。
>
> 误判原因：审查时仅以字符串 `CoreDataStack` 全仓搜索引用，未发现源码命中便判定为死代码。实际上该文件名与类名不一致——`Core/Data/CoreDataStack.swift` 第 5 行声明的类是 `class DataController: ObservableObject`，全仓引用均使用类型名 `DataController` 而非文件名。
>
> 证据：
> - `ControlTrainingApp.swift:7` `@StateObject private var dataController = DataController.shared`，并在第 12 行经 `.environment(\.managedObjectContext, dataController.container.viewContext)` 全局注入；
> - 所有 Repository（`TrainingRepository`/`CheckInRepository`/`PlanRepository`/`ReviewNoteRepository`/`AbilityProfileRepository`）、`ProfileView`、`TrainingViewModel` 等均依赖 `DataController`，是持久化层**唯一实现**。
>
> **结论**：`CoreDataStack.swift` 非死代码，无需删除或整合。仅文件名与类名不一致属可读性瑕疵，可酌情重命名文件为 `DataController.swift` 以提升一致性（建议项，非缺陷）。

### ARC-03 `deleteAllUserData` 存在竞态（reset 早于批量删除完成）

`DataController.deleteAllUserData`：

```swift
performBackgroundTask { context in
    // NSBatchDeleteRequest ...
}
DispatchQueue.main.async {                       // ⚠️ 可能先于上面的后台任务执行
    self?.container.viewContext.reset()
}
```

`performBackgroundTask` 异步派发，`main.async` 在返回后立即入队，二者顺序不保证；`reset()` 可能在批量删除尚未提交时执行，导致数据残留或上下文不一致。
**建议**：在 `performBackgroundTask` 的闭包末尾（保存成功后）再触发 `viewContext.reset()`，或使用 completion 回调串化。

### ARC-04 `AnalysisService.calculateBreathCoordination` 逻辑失效

```swift
let breathRecords = records.filter { $0.methodId != UUID() } // 所有真实记录的 methodId 均 != 零 UUID
let breathCompletion = breathRecords.isEmpty ? 0 : ...
```

该过滤实际等同于"全部记录"，`breathCoordination` 维度并未真正按呼吸训练（`category == .breathing`）计算，与注释/命名不符。属算法正确性缺陷（AC-6.x 评分可信度受影响）。

### ARC-05 导出/导入数据不完整（round-trip 丢数据）

`DataViewModel.collectAllData` 仅导出 `trainingRecords` 与 `checkInRecords`，**遗漏** `CDTrainingPlan` / `CDAbilityProfile` / `CDReviewNote`。导入后用户计划、能力档案、复盘笔记全部丢失；且导入不做去重，重复导入会产生重复记录。

### ARC-06 仓库"写后台 / 读主线程"的瞬时不一致

所有 Repository 写操作经 `performBackgroundTask`（异步），读操作直接用 `viewContext`。虽然 `viewContext.automaticallyMergesChangesFromParent = true`，但合并是异步的，因此"写入后立即读取"的窗口内读不到新数据。典型表现：`HomeViewModel.performCheckIn()` 在 `checkInService.checkIn()` 后立刻 `fetchConsecutiveCheckInDays()`，可能读不到刚写入的打卡，`consecutiveCheckInDays` 不刷新，需下次 `loadData()` 才更新。属体验层缺陷，建议改为写后回调或主线程串行上下文。

### ARC-07 `willEnterForeground` 双重认证触发

`AppDelegate.applicationWillEnterForeground` 调用 `SecurityService.shared.authenticateIfNeeded()`，同时 `SecurityService.configureProtection()` 又注册了 `willEnterForegroundNotification` 观察者调用 `unlockAppIfNeeded()`。二者都会触发 `LAContext` 生物识别弹窗，存在**重复/竞争弹出**风险。注释称"避免双重触发"，但实际仍注册了两套。建议仅保留一处。

## 五、P3 — 次要 / 维护性

- `CoachViewModel.savePartialRecordOnBackground` 在每次 `willResignActive`（如拉出控制中心、来电横幅）且处于训练/暂停态时都会写一条 `isPartial` 记录，可能短时产生多条部分记录，建议做去重/节流。
- `HomeViewModel` / 各 Repository 大量 `print(...)` 错误日志，发布版本应改为 `os_log` 并按环境过滤。
- `fetchActivePlan` 使用 `endDate >= Date()`，历史计划不会展示，符合预期，但无归档入口。
- 通知方法 `scheduleDailyTrainingReminder` 与 `scheduleTrainingReminderWithContent` 共用 identifier `dailyTrainingReminder`，先后调用会互相覆盖，属预期但需注意。
