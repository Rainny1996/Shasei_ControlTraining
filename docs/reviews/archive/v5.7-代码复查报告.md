## 6. 补遗：v5.7 首轮推送后 CI 仍失败（当前轮二次修正）

- 现象：v5.7 首轮 4 文件修复已 `git push`，重跑 "Build APP" **仍 `BUILD FAILED`**，复现 2 处错误：
  - `CoreDataStack.swift:19` — `cannot convert value of type 'String' to expected argument type 'NSObject'`；
  - `SecurityService.swift:307` — `cannot find 'PBKDF2' in scope`（连带 `:312` `cannot infer contextual base in reference to member 'derivedKey'`）。
- 结论修正：
  - `CoreDataStack.swift:19` **并非** GitHub 旧版本问题，而是真实本地错误——`NSPersistentStoreDescription.setOption(_:forKey:)` 的 value 形参为 `NSObject`，Swift `String` 不会隐式桥接为 `NSObject`。修正：`FileProtectionType.complete.rawValue as NSString`。
  - `SecurityService.swift` 的 PBKDF2 **并非**参数标签错误，而是本环境（Xcode 16.4 / iOS 18.5 SDK）的 `CryptoKit` 未将 `PBKDF2` 类型导出到作用域（即便 `import CryptoKit` 存在仍 `cannot find in scope`）。仅改 `deriveKey` 标签无法解决类型不可见问题。修正：改用项目早期已验证可用的 CommonCrypto C API `CCKeyDerivationPBKDF`（补 `import CommonCrypto`，派生 32 字节 PBKDF2-HMAC-SHA256，100,000 次迭代）。
- 当前状态：两处本地修复已落盘，待重跑 CI 验证 `** BUILD SUCCEEDED **`。`PrivacySettingsView` / `DataViewModel` / `CoreDataStack` 括号问题经 v5.7 首轮修复后已不再出现。

## 7. 补遗2：v5.7 二次修正推送后 CI 仍失败（第三次实编译失败 — 自引入回归）

- 现象：v5.7 二次修正（`CoreDataStack:19` 加 `as NSString` + `SecurityService` 改用 CommonCrypto `CCKeyDerivationPBKDF`）已 `git push`，重跑 "Build APP" **仍 `BUILD FAILED`**，仅复现 1 处错误：
  - `SecurityService.swift:317` — `cannot find 'CCPBKDFPRF' in scope`（连带 `:317` `cannot find 'kCCPRFHmacSHA256' in scope`）。
- 根因：二次修正写入了**不存在的 `CCPBKDFPRF(...)` 包装符**（CommonCrypto C API 无此符号）；且本 SDK（Xcode 16.4 / iOS 18.5）的 CommonCrypto **Swift 覆盖层未导出 `kCCPRFHmacSHA256` 常量**（对照同文件的 `kCCPBKDF2` 可正常解析、`kCCSuccess` 可解析，说明覆盖层仅部分导出 C 宏常量）。
- 修复：将第 317 行 `CCPBKDFPRF(kCCPRFHmacSHA256)` 改为原始值 `3`（`kCCPRFHmacSHA256` 的标准 enum 原始值，即 `CCPseudoRandomAlgorithm = 3`），彻底不依赖未导出常量。文件头 `import CommonCrypto` 保留；`CCKeyDerivationPBKDF` / `CCPBKDFAlgorithm(kCCPBKDF2)` / `kCCSuccess` 均确认可解析。
- 影响范围：`CoreDataStack` / `PrivacySettingsView` / `DataViewModel` 经前两轮修复已可编译，本次仅 `SecurityService` 一处回归。
- 当前状态：本地修复已落盘，待重跑 CI 验证 `** BUILD SUCCEEDED **`；通过后跑 `test.yml`（166 用例）。

## 8. 过程缺陷复盘（二次修正自评）

- 二次修正将 CryptoKit `PBKDF2` 替换为 CommonCrypto 的方向正确，但**凭记忆写出了 `CCPBKDFPRF(...)` 这一不存在的包装符**，未核对 CommonCrypto 真实 C 签名；同时误以为 `kCCPRFHmacSHA256` 必定被覆盖层导出。
- **教训**：跨 SDK 使用底层 C API 时，若常量解析失败，应改用其标准原始值而非假设宏可用；关键修复（尤其不可本地编译时）需对照真实 API 签名，`CI 重跑` 仍是唯一放行依据。

## 9. 补遗3：project.yml 嵌入系统框架导致 Embed Frameworks 失败（第四次实编译失败 — 工程配置）

- 现象：前三轮代码修复推送后，"Build APP" **编译阶段已通过**（PBKDF2 等代码错误消失），但**仍 `BUILD FAILED`**，错误变为 `Copy .../Frameworks/UserNotifications.framework ... No such file or directory`（及 Security / LocalAuthentication / CryptoKit / Charts / AVFoundation 共 6 个系统框架）；test 同样因构建失败而 `Testing cancelled`。
- 根因：本地**无 `.pbxproj`**，工程由根目录 `project.yml` 经 **XcodeGen** 在 CI 动态生成。`project.yml` 的 `ControlTraining` target `dependencies` 将 6 个**系统框架**声明为 `framework:` 依赖；XcodeGen 对 `framework:` 依赖**默认 `embed: true`**，生成 “Embed Frameworks” 拷贝阶段，把系统框架按**项目根目录相对路径**复制进 App 包——该路径不存在 → 报错。系统框架随 SDK/设备提供，**不应嵌入**。
- 修复：在 `project.yml` 中为每个 `framework:` 依赖加 `embed: false`（保留链接、取消嵌入）：
  ```yaml
  dependencies:
    - framework: Charts.framework
      embed: false
    - framework: AVFoundation.framework
      embed: false
    - framework: LocalAuthentication.framework
      embed: false
    - framework: CryptoKit.framework
      embed: false
    - framework: Security.framework
      embed: false
    - framework: UserNotifications.framework
      embed: false
  ```
- 影响范围：纯工程配置，**无任何 Swift 源码改动**；链接阶段本就成功，仅移除嵌入拷贝阶段即可消除全部 error。
- 当前状态：`project.yml` 修复已落盘，待重跑 CI 验证 `** BUILD SUCCEEDED **`；通过后跑 `test.yml`（166 用例）。
