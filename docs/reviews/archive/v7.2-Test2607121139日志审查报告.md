## 三、根因分析

### 3.1 `self.repo` 误用(行 89 / 113 / 135)
`Tests需求12直达陪练` 是 `final class`,**没有**名为 `repo` 的实例属性。三个测试方法内部各自声明了局部常量:
```swift
let dc = DataController(inMemory: true)
let repo = PlanRepository(dataController: dc)   // 局部 let,非实例属性
```
但闭包里却写成:
```swift
waitUntil { self.repo.fetchActivePlan() != nil }   // ❌ self.repo 不存在
```
应改为引用局部变量 `repo`(去掉 `self.`)。这是典型的"把局部变量当实例属性"笔误。

### 3.2 `vm.currentPlan?.progress` 为 `Double?`(行 92 / 97 / 118)
`currentPlan` 是 `TrainingPlan?`,因此 `currentPlan?.progress` 的类型是 `Double?`;而 `XCTAssertEqual(_:_:accuracy:)` 第二参 `0.5` 被推断为 `Double`,两边类型不一致触发"cannot convert `Double?` to `Double`"。需将 `progress` 解包为 `Double`。

### 3.3 与 v7.1 报告的关系(重要更正)
v7.1 报告曾将 `tests-需求12` 的 6 条报错判为"**级联假错**,随模型 `id` 默认值修复后应自动消失"。本次核对旧日志 `test-2607121105.log` 的相同文件(行 868/871/873/875/878/880)发现:**这 6 个错误与本次日志逐字相同,完全不受 v7.1 的 `PlanModels.id` 默认值修复影响。**

> ⚠️ **更正 v7.1 的误判**:这 6 个错误是 `tests-需求12` 文件**自身**的真实编译错误(`self.repo` / `Double?`),与 `id` 缺失无关,在两次日志中均无变化。v7.1 的"级联假错"假设不成立,需独立修复。
